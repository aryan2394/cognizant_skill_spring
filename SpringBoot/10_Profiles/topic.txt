Spring automatically sahi configuration load kar dega.

Isi mechanism ko Spring Profiles kehte hain.



What is a Spring Profile?

A Profile is simply the name of an environment in which your application is running.

Examples:

dev
test
staging
prod

Think of it as a label that tells Spring:

"Load the configuration and beans meant for this environment."

For example:

spring.profiles.active=dev

Here, the active profile is dev.

Spring understands:

"I am running in the development environment."

So it loads:

application.properties
+
application-dev.properties

If you change it to:

spring.profiles.active=prod

Spring loads:

application.properties
+
application-prod.properties

No Java code changes are required.

What is the @Profile Annotation?

So far, we've seen profiles used to load different configuration files.

But what if you want to load different Java classes (beans) based on the environment?

That's where @Profile comes in.

It tells Spring:

"Create this bean only if a specific profile is active."

Example

Suppose you have two email services.

Development Email Service
@Service
@Profile("dev")
public class FakeEmailService implements EmailService {

    public void sendEmail() {
        System.out.println("Fake Email Sent");
    }
}
Production Email Service
@Service
@Profile("prod")
public class RealEmailService implements EmailService {

    public void sendEmail() {
        System.out.println("Real Email Sent");
    }
}

If

spring.profiles.active=dev

Spring creates:

FakeEmailService

and ignores

RealEmailService

If

spring.profiles.active=prod

Spring creates:

RealEmailService

and ignores

FakeEmailService
Why is this useful?

Imagine a payment application.

During development, you don't want to charge real money.

So you create:

@Profile("dev")
FakePaymentService

It simply prints:

Payment Successful (Fake)

In production:

@Profile("prod")
RealPaymentService

It actually calls the payment gateway (e.g., Razorpay or Stripe).

Same interface:

public interface PaymentService {
    void pay();
}

Different implementations:

PaymentService
     ↑
 ┌──────────────┐
 │              │
FakePayment   RealPayment
(dev)         (prod)

Spring automatically injects the correct implementation based on the active profile.
