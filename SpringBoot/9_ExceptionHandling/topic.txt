Spring Boot Exception Handling – Summary Notes
Request
GET /api/students/{id}
Host: localhost:8080
Possible Outcomes
Scenario	HTTP Status	Description
Student exists	200 OK	Return student data successfully.
Student not found	404 Not Found	Requested resource does not exist.
Invalid ID format	400 Bad Request	Input/path variable is invalid (e.g., abc instead of 1).
Database is down	500 Internal Server Error	Backend cannot communicate with the database.
Server is down	No HTTP Response	Application is not running (Connection Refused/Timeout).
Backend bug (NullPointerException, etc.)	500 Internal Server Error	Unexpected exception in the application.
Why Exception Handling?

Without exception handling:

Default Spring error response
Stack trace may be exposed
Inconsistent error format
Difficult for frontend to understand

With exception handling:

Meaningful error messages
Consistent response format
Proper HTTP status codes
Better user experience
Professional APIs

1. Normal Request Flow
Client
   ↓
Controller
   ↓
Service
   ↓
Repository

This is the standard Spring Boot architecture.

Client

The client can be

React App
Angular App
Postman
Mobile App
Browser

The client sends an HTTP request.

Example:

GET /students/5
Controller

The request first reaches the controller.

Example

@RestController
@RequestMapping("/students")
public class StudentController {

    @GetMapping("/{id}")
    public StudentResponseDTO getStudent(@PathVariable Long id) {
        return studentService.getStudent(id);
    }
}

The controller's responsibility is

Receive request
Validate request (@Valid)
Call service
Return response

It should not contain business logic.

Service

Now the controller calls the service.

Example

public StudentResponseDTO getStudent(Long id) {

    Student student = studentRepository.findById(id)
            .orElseThrow(() ->
                    new ResourceNotFoundException("Student not found"));

    return mapper(student);
}

The service contains

Business logic
Calculations
Validations
Exception throwing

Notice this line

throw new ResourceNotFoundException("Student not found");

If something goes wrong, the service throws an exception.

Repository

The repository communicates with the database.

Example

public interface StudentRepository
        extends JpaRepository<Student, Long> {
}

Its only responsibility is database operations.

2. What happens when everything is successful?

Suppose

GET /students/5

Student exists.

Flow

Client
   ↓
Controller
   ↓
Service
   ↓
Repository
   ↓
Database
   ↓
Repository
   ↓
Service
   ↓
Controller
   ↓
Client

Response

{
  "id": 5,
  "name": "Aryan",
  "age": 22
}

HTTP Status

200 OK
3. What happens if an error occurs?

Suppose

GET /students/100

Student does not exist.

Repository

↓

returns

Optional.empty()

Service

↓

throws

throw new ResourceNotFoundException("Student not found");

Now notice...

The controller never completes.

Instead, the exception is sent to the Exception Handler.

Client
   ↓
Controller
   ↓
Service
   ↓
Repository
        ↑
Exception
        ↑
Exception Handler
        ↓
Client

That's exactly what the box below the controller in your screenshot represents.

4. Why do we need an Exception Handler?

Without one, Spring Boot returns something like

{
  "timestamp":"...",
  "status":500,
  "error":"Internal Server Error",
  "trace":"very long stack trace..."
}

This is

ugly
inconsistent
exposes internal details

Instead, we want

{
    "message":"Student not found",
    "status":404
}

or

{
    "timestamp":"2026-07-09T10:00:00",
    "status":404,
    "message":"Student not found"
}

This looks much more professional.
