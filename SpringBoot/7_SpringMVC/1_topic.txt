The client sends an HTTP request:

GET /users

Tomcat receives the request and creates:
HttpServletRequest
HttpServletResponse

Tomcat forwards the request to the appropriate servlet:
UserServlet
OrderServlet
PaymentServlet

The selected servlet processes the request and returns an HTTP response.

Typical responsibilities of a traditional Servlet:
Request parameter checking (validation)
Converting JSON into Java objects (deserialization)
Processing business logic
Sending the HTTP response

This is the traditional Servlet architecture that Spring MVC improves by introducing a single DispatcherServlet instead of having many individual servlets like UserServlet, OrderServlet, and PaymentServlet.

## What is HandlerMapping?

**HandlerMapping** is a Spring MVC component that tells the **DispatcherServlet** which controller method should handle an incoming request.

Think of it as a **routing table**.

### Example

Suppose you have:

```java
@RestController
@RequestMapping("/student")
public class StudentController {

    @GetMapping
    public List<Student> getAllStudents() {
        // ...
    }

    @PostMapping
    public Student createStudent(@RequestBody Student student) {
        // ...
    }
}
```

Spring internally creates mappings like:

| HTTP Request    | Controller Method                    |
| --------------- | ------------------------------------ |
| `GET /student`  | `StudentController.getAllStudents()` |
| `POST /student` | `StudentController.createStudent()`  |

This mapping information is stored by **HandlerMapping**.

### Request Flow

```text id="t78vmz"
Client
   │
   ▼
GET /student
   │
   ▼
DispatcherServlet
   │
   ▼
HandlerMapping
   │
   └── Finds:
       GET /student
       ↓
       StudentController.getAllStudents()
   │
   ▼
Controller Method Executes
   │
   ▼
Response Returned
```

### Internally

When your application starts, Spring scans annotations like:

* `@RequestMapping`
* `@GetMapping`
* `@PostMapping`
* `@PutMapping`
* `@DeleteMapping`

and builds a lookup table similar to:

```text id="lcjlwm"
/student   + GET    -> StudentController.getAllStudents()
/student   + POST   -> StudentController.createStudent()
/student/{id} + GET -> StudentController.getStudentById()
/student/{id} + PUT -> StudentController.updateStudent()
/student/{id} + DELETE -> StudentController.deleteStudent()
```

Then, whenever a request arrives, the **DispatcherServlet** asks **HandlerMapping**:

> "Which controller method should handle this request?"

HandlerMapping returns the correct method, and the DispatcherServlet invokes it.

### One-line definition

> **HandlerMapping is the Spring MVC component that maps an incoming HTTP request (URL + HTTP method) to the appropriate controller method.**

THese is the final structure and flow of Spring MVC 

1. Tomcat → DispatcherServlet

The client sends an HTTP request.
Tomcat receives it and forwards it to the DispatcherServlet.

2. DispatcherServlet → HandlerMapping

The DispatcherServlet asks:

"Which controller should handle this request?"

3. HandlerMapping → DispatcherServlet

HandlerMapping looks at the URL and HTTP method.
It returns the matching controller method.

Example:

GET /users
        ↓
UserController.getAllUsers()

4. DispatcherServlet → Controller

The DispatcherServlet invokes the appropriate controller method.

Example:

@GetMapping("/users")
public List<User> getUsers() {
    return service.getUsers();
}

5. Controller → DispatcherServlet

The controller executes the business logic (often through a service layer).
It returns a result (such as a Java object, list, or view name) to the DispatcherServlet.

6. DispatcherServlet → Tomcat → Client

The DispatcherServlet converts the result into an HTTP response (for example, JSON in a REST API).
It gives the response to Tomcat.
Tomcat sends the final HTTP response back to the client.
