sabse pahle we have to craete interceptor i.e  we have to implement HandlerInterceptor and override preHandle, postHandle, afterCompletion methods.


@Component
//taaki spring can manage its bean
public class LoggingInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) throws Exception {

        System.out.println("preHandle() executed");
        return true;    // Continue request processing
    }

    @Override
    public void postHandle(HttpServletRequest request,
                           HttpServletResponse response,
                           Object handler,
                           ModelAndView modelAndView) throws Exception {

        System.out.println("postHandle() executed");
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler,
                                Exception ex) throws Exception {

        System.out.println("afterCompletion() executed");
    }
}


and then we have to configure this interceptor in WebMvcConfigurer's addInterceptors method.

@Configuration
public class WebConfig implements WebMvcConfigurer {
    LoggingInterceptor loggingInterceptor;
    public WebConfig(LoggingInterceptor interceptor)
    {
        this.loggingInterceptor=interceptor;
    }
    @Override
    public void addInterceptors(InterceptorRegistry registry)
    {
        registry.addInterceptor(loggingInterceptor);
    }
}

Why do we need WebMvcConfigurer for Interceptors?

An interceptor is not a Servlet component.

It belongs to Spring MVC, not the Servlet API.

When you write:

@Component
public class LoggingInterceptor implements HandlerInterceptor {
}

Spring only creates the bean.

It doesn't know:

Which URLs should use it?
Should it apply to every request?
Should /login be excluded?
In what order should multiple interceptors run?

So you must explicitly register it.

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Autowired
    private LoggingInterceptor interceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {

        registry.addInterceptor(interceptor);
    }
}

Now Spring MVC knows:

"Whenever a request reaches the DispatcherServlet, execute this interceptor."

Without WebConfig
@Component
LoggingInterceptor
        │
        ▼
Spring creates the object

❌ Never attached to Spring MVC

The interceptor bean exists, but it is never used.

With WebConfig
@Component
LoggingInterceptor
        │
        ▼
Spring Bean
        │
        ▼
WebMvcConfigurer
        │
        ▼
InterceptorRegistry
        │
        ▼
DispatcherServlet
        │
        ▼
Controller

Now every matching request passes through the interceptor.

Why is manual registration useful?

Suppose you have three interceptors:

LoggingInterceptor
AuthInterceptor
PerformanceInterceptor

You can register them like this:

@Override
public void addInterceptors(InterceptorRegistry registry) {

    registry.addInterceptor(loggingInterceptor);

    registry.addInterceptor(authInterceptor)
            .addPathPatterns("/api/**");

    registry.addInterceptor(performanceInterceptor)
            .excludePathPatterns("/login");
}

Spring now knows:

LoggingInterceptor → all requests
AuthInterceptor → only /api/**
PerformanceInterceptor → all except /login

This flexibility is why Spring MVC requires explicit registration.


1. HttpServletRequest request

This object contains everything that came from the client.

Example request:

POST /students
Authorization: Bearer xyz123
Content-Type: application/json

{
   "name":"Aryan",
   "age":22
}

Using request, you can access:

URL
request.getRequestURI();

Output

/students
HTTP Method
request.getMethod();

Output

POST
Headers
request.getHeader("Authorization");

Output

Bearer xyz123


2. HttpServletResponse response

This represents the response that will be sent back to the client.

You can modify it.

Example:

response.setStatus(401);

Response becomes

401 Unauthorized


3. Object handler

This is one of the most important parameters.

It represents the object that will handle the request.

Usually it is a HandlerMethod, which contains information about:

Controller class
Controller method
Method annotations

Suppose you have

@RestController
@RequestMapping("/students")
public class StudentController {

    @GetMapping("/{id}")
    public Student getStudent() {

    }
}

When the request comes

GET /students/5

handler points to

StudentController.getStudent()

Since handler is declared as Object, you normally cast it:

if (handler instanceof HandlerMethod hm) {

    System.out.println(hm.getMethod().getName());

    System.out.println(hm.getBeanType().getSimpleName());
}

Output

getStudent
StudentController

This is useful for logging, auditing, or checking custom annotations.

4:ModelAndView modelAndView

Available only in postHandle().

postHandle(..., ModelAndView modelAndView)

This is mainly used in Spring MVC applications that return HTML views.



5. Exception ex

Available only in afterCompletion().

afterCompletion(..., Exception ex)

If the controller throws an exception:

@GetMapping
public Student getStudent(){

    throw new RuntimeException("Database Error");
}

then

ex

contains that exception.

You can log it:

if(ex != null){

    System.out.println(ex.getMessage());
}

Output

Database Error

If no exception occurs:

ex == null

1. Interceptor Register Kahan Hota Hai?

Ek configuration class banate hain jo WebMvcConfigurer implement karti hai.

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Autowired
    private LoggingInterceptor loggingInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(loggingInterceptor);
    }
}

Yahi Spring ko batata hai ki kaunsa interceptor use karna hai.

2. addInterceptor()
registry.addInterceptor(loggingInterceptor);

Meaning:

DispatcherServlet ke through jo bhi request jayegi, usme ye interceptor execute hoga.

Agar kuch aur specify nahi karoge to

/**
 * Applies to every controller request
 */

Matlab

GET  /students
POST /students
PUT  /students/1
DELETE /students/1

sab par chalega.

3. addPathPatterns()

Sirf selected URLs par interceptor chalana ho.

registry.addInterceptor(loggingInterceptor)
        .addPathPatterns("/students/**");

Iska matlab

✔

/students
/students/1
/students/create
/students/update/5

Par chalega.

❌

/users
/products
/login

Par nahi chalega.

4. Multiple Path Patterns
registry.addInterceptor(loggingInterceptor)
        .addPathPatterns("/students/**",
                         "/teachers/**");

Dono modules par chalega.

5. excludePathPatterns()

Kuch URLs ko ignore karna ho.

registry.addInterceptor(loggingInterceptor)
        .excludePathPatterns("/login");

Matlab

/login

par interceptor execute nahi hoga.

Example

registry.addInterceptor(loggingInterceptor)
        .addPathPatterns("/**")
        .excludePathPatterns("/login",
                             "/register");

Ye sabse common interview example hai.

6. Order of Multiple Interceptors
registry.addInterceptor(authInterceptor);

registry.addInterceptor(loggingInterceptor);

registry.addInterceptor(rateLimitInterceptor);

Execution

preHandle()

Auth
↓

Logging
↓

RateLimit
↓

Controller

Response aate waqt reverse order hota hai

Controller

↓

RateLimit postHandle()

↓

Logging postHandle()

↓

Auth postHandle()

Same rule

afterCompletion()

me bhi follow hota hai.

7. order()

Execution order manually set kar sakte ho.

registry.addInterceptor(authInterceptor)
        .order(1);

registry.addInterceptor(loggingInterceptor)
        .order(2);

Smaller order pehle execute hota hai.
